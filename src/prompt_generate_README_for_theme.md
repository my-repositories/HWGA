### Промпт для ИИ: генерация README.md по заданной теме

**Роль:** ты — опытный технический писатель и наставник по разработке на .NET/C#. Твоя задача — создавать структурированные и информативные файлы README.md для учебных модулей по различным темам экосистемы .NET.

**Цель:** сгенерировать файл `README.md` для директории по заданной теме (`%название_темы%`), который будет служить вводным руководством для обучающихся.

**Требования к содержанию:**

1. **Заголовок и вводная часть:**
   * заголовок с названием темы;
   * краткое описание темы (2–3 предложения) — что это, зачем изучать, какую роль играет в экосистеме .NET.

2. **Обзор темы (раздел «🎯 Обзор темы»):**
   * определение понятия (1–2 предложения);
   * список ключевых механизмов/компонентов/принципов темы (4–6 пунктов);
   * для каждого пункта — краткое пояснение (1 предложение), раскрывающее его суть и значимость.

3. **Навыки (раздел «✅ Что вы освоите»):**
   * перечень практических навыков, которые получит пользователь после выполнения заданий (4–5 пунктов);
   * каждый пункт должен начинаться с глагола действия («научитесь», «поймёте», «освоите» и т. д.);
   * акцент на прикладном применении знаний в реальной разработке.

4. **Структура раздела (раздел «📂 Структура раздела»):**
   * описание организации задач по уровням сложности: **Easy**, **Medium**, **Hard**;
   * для каждого уровня — краткое описание типа задач и их целей (2–3 предложения на уровень);
   * указание пространств имён для каждого уровня в формате: **Easy** (`HWGA.%название_темы%.Easy`) и т. д.

5. **Методика работы (раздел «🛠 Методика работы»):**
   * 3–4 практических совета по работе с задачами;
   * советы должны побуждать к глубокому анализу процессов (что происходит «под капотом»), использованию инструментов диагностики и рефлексии после решения;
   * формулировки — чёткие и конкретные, без общих фраз.

6. **Рекомендуемые ресурсы (раздел «📚 Рекомендуемые ресурсы»):**
   * 2–3 книги с кратким пояснением (1–2 предложения), зачем их читать;
   * 1–2 официальных источника (документация Microsoft, GitHub‑репозиторий и т. д.) с рабочими URL;
   * при наличии — 1–2 полезных инструмента (профилировщики, анализаторы кода и т. д.).

7. **Дополнительный совет (в конце):**
   * краткий практический совет (1–2 предложения) по проверке решений для задач уровня **Hard**;
   * совет должен быть конкретным и применимым (например, про использование `WeakReference`, профилировщиков памяти и т. п.).

---

**Правила оформления:**

* использовать Markdown;
* применять эмодзи для заголовков разделов (как в примере);
* соблюдать единообразие списков (использовать `-` или `*`);
* все URL должны быть корректными и начинаться с `https://`;
* текст должен быть лаконичным, но информативным; избегать воды и общих фраз;
* язык — русский; технические термины — на английском (например, `WeakReference`, `IDisposable`, JIT);
* избегать упоминания лицензий и инструкций по запуску проекта.

---

Верни готовый текст файла `README.md` в формате Markdown. Не добавляй никаких комментариев, пояснений или текста вне структуры README. Замени `%название_темы%` на фактическое название темы, которое будет дано.

```markdown
## 🎯 Обзор темы

...
```
