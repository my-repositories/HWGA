## JitCompilationAndOptimization (Hard)

**Задача:** Создать бенчмарк для сравнения производительности кода до и после JIT‑компиляции в CLR. Реализовать два алгоритма сортировки (простой и оптимизированный), измерить время выполнения до и после «прогрева» JIT, проанализировать влияние оптимизаций.

**Пример:**

Вход:
```csharp
var data = GenerateRandomArray(10_000);
RunBenchmark(data, new BubbleSort(), "Bubble Sort");
RunBenchmark(data, new QuickSort(), "Quick Sort");
```

Выход:
```
=== Bubble Sort ===
First run (cold): 520 ms
After JIT (hot): 480 ms
Speedup: 7.7%

=== Quick Sort ===
First run (cold): 15 ms
After JIT (hot): 8 ms
Speedup: 46.7%
```

Объяснение: Первый запуск («холодный») включает время JIT‑компиляции. Последующие запуски («горячие») используют скомпилированный код, что даёт прирост производительности. Оптимизированный алгоритм (Quick Sort) выигрывает больше от JIT.

**Подсказка:** Используйте `System.Diagnostics.Stopwatch` для замеров времени. Проведите несколько итераций для стабилизации результатов. Пространство имён: `namespace HWGA.CLR.Hard.JitCompilationAndOptimization;`.

**Ограничения:**
* Реализовать `BubbleSort` и `QuickSort` как классы, реализующие интерфейс `ISorter`.
* Замерять время до и после минимум 5 «прогревочных» запусков.
* Усреднять результаты по 10 итерациям для «горячего» теста.
* Размер массива — не менее 10 000 элементов.
* Код должен быть совместим с .NET 6+ и не использовать внешние библиотеки бенчмарков.

**Что проверить в решении:**
* Корректность сортировки (массив упорядочен после вызова).
* Достоверность замеров времени (разница между «холодным» и «горячим» запуском).
* Анализ влияния JIT: оптимизированные алгоритмы должны показывать больший прирост.
* Обработка крайних случаев (пустой массив, отсортированный массив, дубликаты).
